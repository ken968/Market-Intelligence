MARKET INTELLIGENCE SYSTEM: COMPREHENSIVE ARCHITECTURE & RESEARCH WHITEPAPER
PART 2: DATA ENGINEERING & LOOK-AHEAD BIAS MITIGATION

1. IN-PROCESS COLUMNAR ANALYTICS (DUCKDB + POLARS)
To handle the memory overhead and multi-threading bottlenecks inherent in pandas-based CSV parsing within a Streamlit environment, the system utilizes DuckDB. DuckDB is a serverless, embedded RDBMS optimized for Online Analytical Processing (OLAP).
- Vectorized Query Execution: DuckDB processes data in columnar vectors, maximizing CPU cache locality and drastically reducing query latency compared to row-based iteration.
- Zero-Copy Integration: Through Apache Arrow memory mapping, the system transitions data from DuckDB relations directly into Polars DataFrames (`rel.pl()`). Polars, written in Rust, provides multi-threaded lazy evaluation, allowing complex aggregations to occur with virtually zero memory copying overhead.

2. ATOMICITY AND DUAL-WRITE FAIL-SAFE MECHANISMS
Financial terminals cannot afford silent failures during data ingestion. The `MarketDataStore` class implements a Dual-Write architecture simulating Pseudo-ACID properties.
- Concurrency Control: Read operations are strictly executed with `read_only=True` to prevent blocking the UI during background data ingestion.
- Fallback Redundancy: Write operations execute `CREATE OR REPLACE TABLE` in DuckDB while simultaneously exporting a `.csv` backup. If an OS-level file lock prevents the DuckDB write, the Exception is caught, the CSV is updated, and the system gracefully degrades to reading the CSV, ensuring zero downtime. (Note: Phase 6 SRE roadmaps include timestamp-based auto-healing to resynchronize the DB from the CSV post-lock).

3. CAUSAL INTEGRITY & LOOK-AHEAD BIAS PREVENTION
In quantitative finance, Look-Ahead Bias (Data Leakage) is the most pervasive flaw, occurring when a model inadvertently trains on data that was not publicly available at the theoretical time of prediction.
- Strict Forward-Filling (ffill): Consider the CFTC COT report, which aggregates data as of Tuesday but is not released to the public until Friday evening. The pipeline rigidly indices this data to the Friday release date. The values are then propagated forward (`ffill()`) to Saturday, Sunday, and the subsequent week.
- Prohibition of Interpolation: Linear interpolation between data points is strictly prohibited in the pipeline. Interpolating between a Tuesday and a Friday essentially "leaks" the Friday trajectory into Wednesday and Thursday, violating temporal causality.
- Macroeconomic Lagging: Economic indicators (e.g., CPI, NFP) are subjected to automated lagging ($t-3$, $t-6$ months) within the XGBoost feature generation step to simulate the reality of delayed government reporting schedules and retrospective revisions.
